我现在需要开发一个校园互助平台的后端部分 作为微信小程序 采用微服务架构 

具体业务需求：

以下是基本服务

1. **API Gateway (网关服务)**：统一入口，HTTP 路由，JWT 鉴权，限流，跨域，协议转换。

2. **User Service (用户服务)**：微信登录、RBAC 权限、选择学校)。

3. **Content Service (内容服务)**：通用帖子、**失物招领**、**二手交易**、评论、点赞、收藏、ES 数据同步。

4. **Task Service (任务服务)**：**跑腿代拿**、**组队/拼车**。处理复杂的订单状态机、悬赏资金冻结、抢单并发控制、超时取消。

5. **Message Service (消息服务)**：站内信、系统/互动/业务通知分发、WebSocket 长连接管理。

6. **Admin Service (管理服务)**：内容审核、用户封禁、数据看板、报表统计。

7. **File Service (文件服务)**：图片/文件上传、图片压缩。

   不同学校要展示不同内容

#### 1. 校园特色场景化模块（细分互助类型）

通用的“帖子”无法满足具体的互助需求，建议将帖子抽象为不同的**业务模板**：

- **失物招领**：增加特定字段（丢失/拾取地点、物品分类、特征描述），支持对接校园地图 API 标记位置。
- **跑腿代拿/代办**：引入**悬赏机制**和**状态流转**（发布 -> 被接单 -> 进行中 -> 确认完成 -> 评价）。需要处理超时未接单自动取消的逻辑。
- **二手交易**：增加价格、成色、交易方式（面交/快递）。可引入简单的“意向金”或“担保交易”概念（后期拓展）。
- **组队/拼车/自习**：引入**人数限制**和**报名审核**机制（如：拼车去机场，限4人，满员自动结贴）

#### 2. 深度互动与消息中心

- 消息通知系统

  ：

  - **系统通知**：帖子审核结果、违规警告、系统公告。
  - **互动通知**：评论、回复、点赞、新增粉丝。
  - **业务通知**：您的跑腿订单已被接单、对方已确认完成。



技术栈为gin,mysql,redis,grpc,etcd,rabbitmq,Elasticsearch，Tracing (Jager),Logging(日志要与链路以及消息队列结合)



###  开发演进路线 (Roadmap)

1. Phase 1：基建与骨架 (Week 1)
   - 搭建 MySQL/Redis/RabbitMQ/etcd/ES/Jaeger 环境 。
   - 完成服务发现、统一日志(带TraceID)、MQ封装。
   - 跑通 Gateway -> gRPC -> 数据库的最简链路，并在 Jaeger 看到完整 Span。
2. Phase 2：用户与内容闭环 (Week 2)
   - 完成 User Service (登录/鉴权)。
   - 完成 Content Service (发帖/失物/二手)，接入 DFA 敏感词过滤。
   - 实现 ES 数据同步与复合搜索接口。
3. Phase 3：任务与交易流转 (Week 3)
   - 完成 Task Service，实现跑腿状态机。
   - 攻克难点：Redis/etcd 分布式锁实现抢单，RabbitMQ 延迟队列实现超时取消。
4. Phase 4：消息中心与可观测性完善 (Week 4)
   - 完成 Message Service，实现 WebSocket 推送和离线消息拉取。
   - 全面测试 MQ 的 TraceID 透传，确保 Jaeger 链路不断层。
   - 验证日志通过 MQ 异步收集并在 Loki/Kibana 中通过 TraceID 检索。
5. Phase 5：管理后台与压测调优 (Week 5)
   - 完成 Admin Service 和数据统计。
   - 使用 JMeter 或 wrk 对核心接口（如抢单、发帖）进行压测，根据 Prometheus 指标调优连接池、协程池和 Redis 缓存策略。
